# SLASH — A modern shell with Unix roots

> A clean, fast, modern shell written in Zig. Not a programming language.
> Not a framework. Not a compatibility layer. A shell.

---

## Vision

Every existing shell is compromised. Bash is a 1989 design patched for 37 years.
Zsh is bash with more footguns and a plugin ecosystem that papers over them. Fish
is friendly but timid. Nushell forgot it was supposed to be a shell and became a
programming language. PowerShell is an abomination.

Slash is none of these things.

Slash is a shell whose only job is to run commands and compose them. It does that
job with precision, expressiveness, and speed. It has exactly enough language
features to make shell scripting not miserable — variables, simple loops, simple
conditionals, inline math — and no more. When you need a real programming
language, you call one. Slash gets out of the way.

Slash is written entirely in Zig. It uses a grammar-driven parser generated by the
same Zig-based grammar engine used in the `em` MUMPS runtime. It stores history
and navigation state in SQLite. It has a rich, fast, async prompt. It does
syntax highlighting live as you type. It handles job control correctly — not
approximately, not mostly, but correctly.

It will be awesome.

---

## Core Philosophy

**The shell's job is composition, not computation.**

- Run commands
- Compose them with pipes and redirects
- Manage processes and jobs
- Navigate the filesystem quickly
- Remember what you have done
- Get out of your way

If you need string manipulation, data structures, or complex logic: use a real
language. Write a script in TypeScript, Rip, Python, or Perl, and call it from
Slash. The shell executes it. The shell does not replace it.

**One syntax, two forms.**

Slash supports both brace-delimited blocks (for one-liners at the prompt) and
indentation-delimited blocks (for scripts). They are identical in semantics.
The lexer handles the difference. The grammar sees the same tokens either way.

At the prompt, extend a line with `\` at the end. In a `.slash` script file,
indentation alone delimits blocks — no `\` needed.

**No configuration required for a great experience.**

Slash ships with sane defaults for everything. The prompt is rich out of the box.
History works out of the box. Directory jumping works out of the box. You can
customize everything, but you do not have to customize anything.

## Implementation Language: Zig

Slash is written in Zig, no exceptions. Reasons:

- **No garbage collector.** A shell cannot pause. Ever. Zig has no GC.
- **No runtime.** The binary is the shell. No VM, no interpreter, no JIT warmup.
- **Instantaneous startup.** Shells launch constantly — in scripts, subshells,
  terminal multiplexers. Every millisecond matters.
- **Direct syscall access.** `fork`, `execve`, `waitpid`, `tcsetpgrp`, `setpgid`,
  `kill` — all via `std.posix`. No libc wrappers doing unexpected things. Full
  visibility and control over every system interaction.
- **The team already lives in Zig.** The `em` MUMPS runtime and its grammar engine
  are Zig. Slash reuses that investment directly.

## Parser: Zig Grammar Engine

Slash uses the same Zig-based grammar engine developed for `em`. This is not
negotiable and not a compromise — it is an advantage.

Most shells have hand-rolled parsers written by systems programmers who find
parsing painful. Slash's parser is generated from a clean formal grammar. This
means:

- The grammar is the documentation
- Grammar changes propagate automatically to the parser
- Live syntax highlighting uses the same parser, not a pile of regexes
- Error messages are precise and grammar-derived
- The shell grammar is small — far simpler than MUMPS — so the parser is fast

**Slash does not construct an AST.** The parser outputs s-expressions directly —
this is the native output format of the Solar/Zig grammar engine (`mode = 'sexp'`).
The executor walks s-expressions recursively. No AST node types, no visitor
pattern, no intermediate representation. What the parser produces is what the
executor consumes.

```
input text  →  lexer  →  tokens  →  parser  →  s-expressions  →  executor
```

S-expressions are trivially walkable: the executor pattern-matches on the head
of each list and dispatches. `["pipe", a, b]` — run `a`, wire to `b`, run `b`.
`["if", cond, then, else]` — evaluate `cond`, branch. Simple, fast, debuggable.

The `--sexpr` flag dumps the parsed s-expression for any command — invaluable
during development and when debugging unexpected behavior.

The grammar file will be developed as a separate deliverable. The grammar engine
accepts it and generates the parser. The shell is built on top of that parser.

## SQLite Integration

Slash embeds SQLite as its persistent state store. SQLite is used for:

**Command history** — every command is stored with timestamp, working directory,
exit code, and duration. History is searchable with real queries. Duplicates are
deduplicated intelligently. History is shared across all running Slash instances
in real time.

**Directory frecency** — every `cd` is recorded. Slash tracks visit frequency and
recency to rank directories for fast jumping. The last N visited directories are
instantly accessible by number. The ranking algorithm is: `frequency * recency_weight`.

**No flat files.** No `~/.slash_history`. No `~/.slash_dirs`. One database.
One schema. Queryable, transactional, reliable.

## The Prompt

The Slash prompt is rich, fast, and async. It never blocks. All slow operations
(git status, remote branch state) run in the background and update the prompt
when complete without disrupting what the user is typing.

**Default prompt elements (left side):**

```
~/projects/slash  main ✓  zig 0.14  3ms  $
```

- Current directory (abbreviated with `~`)
- Git branch and dirty state (async, non-blocking)
- Current language version if relevant (zig, node, python — detected from project files)
- Last command duration (shown when > 1ms)
- Exit code indicator (green `$` on success, red `$` with code on failure)

**Right prompt:**

```
                                          Thu 14:23
```

- Time (updates between commands, not while typing)

**Prompt is fully configurable** via a simple configuration format. The default
is good enough that most users will never touch it.

## Directory Navigation

Directory navigation is a first-class feature, not a plugin.

```
cd projects/slash    # go there, record it
..                   # go up (no need to type 'cd ..')
...                  # go up two levels
j slash              # fuzzy jump to most frecent match for 'slash'
```

**Quick directory jumping** — press `Esc-=` (default binding, configurable)
and Slash shows the most recent directories below the prompt:

```
  1  ~/projects/slash       (moments ago)
  2  ~/projects/em          (2 hours ago)
  3  ~/projects/solar       (yesterday)
  4  ~/Documents            (3 days ago)
  5  /tmp                   (last week)
```

Type the number — Slash jumps immediately. No Enter needed. Press Escape to
cancel. This is backed by SQLite frecency data. Fast, accurate, zero
configuration.

**File-aware cd** — if you `cd` to a file, Slash goes to its parent
directory instead of printing an error. Enabled by default.

```
cd ~/projects/slash/src/main.zig    # goes to ~/projects/slash/src/
```

Disable with `set file-aware-cd false`.

**Quick navigation hotkeys** — also built-in defaults:

```
Esc-1              # up 1 directory
Esc-2              # up 2 directories
Esc-3              # up 3 directories
Esc--              # previous directory (cd -)
Esc-l              # ls -la
```

## Syntax Highlighting

Slash does live syntax highlighting as you type, using the actual parser — not
regexes. This means the highlighting is always correct, even for complex
multi-line commands.

**Color scheme (defaults, all configurable):**

- Command names: **bright white** — turns green if the command exists in PATH,
  red if it does not
- Flags and options: cyan (`-la`, `--verbose`)
- Strings: yellow (`"hello world"`)
- Variables: magenta (`$HOME`, `$name`)
- Globs: blue (`*.zig`, `/tmp/*.log`)
- Operators: white (`|`, `&&`, `>`)
- Keywords: bright blue (`if`, `for`, `while`, `unless`, `until`)
- Numbers: cyan (`42`, `3.14`)
- Regex literals: yellow-green (`/\.zig$/`)
- Errors: red underline (unknown command, syntax error)

Highlighting works on multi-line commands. When you use `\` to continue a line
at the prompt, the highlight context carries across lines correctly.

## Language Features

Slash has exactly the language features a shell needs. No more.

### Commands

The fundamental unit. A command is a program name followed by arguments.

```
ls -la /tmp
git commit -m "initial commit"
zig build test
```

### Pipelines

Commands composed with pipes. The exit code of the last command is the exit
code of the pipeline.

```
ls -la | grep zig | sort
cat errors.log | grep -v DEBUG | head -20
```

Pipe stderr explicitly:

```
make |& grep error
```

### Redirections

```
command > out.txt          # stdout to file
command >> out.txt         # stdout append
command 2> err.txt         # stderr to file
command > out.txt 2>&1     # stdout and stderr to file
command &> out.txt         # stdout and stderr to file (shorthand)
command < input.txt        # stdin from file
command '''                # heredoc (literal)
    line one
    line two
    '''
command """                # heredoc (interpolated)
    hello $name
    """
command <<< "hello"        # herestring
```

### Boolean Operators

Both symbol and word forms are always valid and identical in meaning.

```
make && echo done          # run if previous succeeded
make and echo done         # identical

make || echo failed        # run if previous failed
make or echo failed        # identical

! test -f foo              # negate
not test -f foo            # identical

cmd1 xor cmd2              # exactly one must succeed (word form only)
```

### Semicolons

Semicolons separate commands on one line. Both commands run regardless of
exit code. Different from `&&` (conditional) and `&` (background).

```
mkdir build; cd build
echo starting; make; echo done
```

### Background Jobs

```
long-running-thing &       # run in background
make & ./watch-logs        # two things at once
```

### Subshell Grouping

Parentheses run commands in a subshell. State changes (variables, `cd`)
do not affect the parent.

```
(cd /tmp; tar xzf archive.tgz)    # parent stays in original directory
(long-thing &) &> /dev/null       # fork, background, detach, silence
```

### Comments

`#` starts a comment anywhere — in scripts and at the interactive prompt.

```
ls -la    # show everything
# this whole line is a comment
```

### Variables

Simple assignment. No type declarations. Uppercase variables are automatically
exported to child processes — no `export` keyword. This is not a convention,
the shell enforces it.

```
name = "steve"
count = 42
PATH = "$PATH:/usr/local/bin"
files = *.zig
```

Variable expansion:

```
echo $name
echo "Hello $name"
echo "Count: $count"
```

Remove a variable with `-`:

```
name = -                   # unset (bare - means remove)
name = "-"                 # set to the string "-"
```

### Inline Math

If an expression is unambiguously arithmetic, Slash evaluates it. No `$(( ))`
syntax. No calling `bc`. Just math.

```
x = 10 + 4               # x is 14
y = $x * 3               # y is 42
head -$(10 + 4) file.txt  # head -14 file.txt
echo $((100 / 7))         # 14 (integer division)
```

The rule: if the left side of `=` is a word and the right side is a numeric
expression, it is math. If you type `1 + 2 * 8` as a command, Slash evaluates it
and prints `17`. This is not ambiguous — no program is named `1`.

Supported operators: `+`, `-`, `*`, `/`, `%`, `**` (power), parentheses for
grouping.

### Special Variables

| Variable | Meaning |
|----------|---------|
| `$?` | Exit code of the last command |
| `$$` | PID of the current Slash process |
| `$!` | PID of the most recent background job |
| `$#` | Number of positional arguments |
| `$*` | All positional arguments |
| `$0` | Name of the current script (or "slash" in interactive mode) |

These are read-only. They are always available.

### Conditionals: if / unless

Conditions are either pipeline exit codes or explicit comparisons.

```
# Exit code based
if git diff --quiet { echo clean }
unless git diff --quiet { echo dirty }

# Comparison based
if $count == 0 { echo "nothing found" }
if $name != "root" { echo "not root" }
if $size >= 1000 { echo "large file" }
unless $errors == 0 { echo "problems" }

# With else
if git diff --quiet {
    echo clean
} else {
    echo dirty
}

# Chained
if $x == 1 {
    echo one
} else if $x == 2 {
    echo two
} else {
    echo other
}
```

**Comparison operators:** `==`, `!=`, `<`, `>`, `<=`, `>=`

Slash is smart about comparison types: if both sides are numeric, numeric
comparison is used. Otherwise string comparison. No `-eq` vs `=` nonsense.

**File tests** — use `test` with flags. No `[`, `[[`, or `]]` syntax.

```
if test -f $file { echo "it's a file" }
if test -d $path { echo "it's a directory" }
unless test -e $path { echo "doesn't exist" }
```

| Flag | Meaning |
|------|---------|
| `-e` | Path exists |
| `-f` | Is a regular file |
| `-d` | Is a directory |
| `-s` | Exists and is non-empty |
| `-r` | Is readable |
| `-w` | Is writable |
| `-x` | Is executable |
| `-L` | Is a symbolic link |

`test` is a Slash builtin — fast, no external process.

**Regex matching:**

```
if $file =~ /\.zig$/ { echo "zig source" }
if $name !~ /^test/ { echo "not a test" }
```

Regex uses `/pattern/` delimiter with optional flags: `/pattern/i` for
case-insensitive. The regex engine is Oniguruma via Zig FFI — battle-tested,
full-featured, supports PCRE-style patterns.

**Boolean logic in conditions:**

```
if $x > 0 and $x < 100 { echo "in range" }
if $name == "foo" or $name == "bar" { echo "matched" }
if not $file =~ /\.tmp$/ { echo "not a temp file" }
if ($a == 1 or $b == 2) and $c == 3 { echo "complex" }
```

### Loops: for / while / until

```
# For over a glob
for f in *.zig {
    echo $f
}

# For over a list
for name in alice bob carol {
    echo "Hello $name"
}

# While a command succeeds
while server-is-running {
    sleep 1
}

# While a condition holds
while $count < 10 {
    echo $count
    count = $count + 1
}

# Until a condition becomes true (inverse of while)
until $done == 1 {
    sleep 1
}
```

`break` exits the innermost loop. `continue` skips to the next iteration.

```
for f in *.zig {
    if $f == "test.zig" { continue }
    if $f == "stop.zig" { break }
    echo $f
}
```

### Pattern Matching: try

`try` matches a value against strings, regexes, or a catch-all. It is the
clean replacement for shell `case` statements.

```
try $action
    "start"    { npm run dev }
    "build"    { npm run build }
    "test"     { npm run test }
    else       { echo "usage: react <start|build|test>" }
```

Regex patterns use `/pattern/` delimiters with optional flags:

```
try $file
    /\.zig$/       { echo "zig source" }
    /\.rs$/        { echo "rust source" }
    /test/i        { echo "test file" }
    else           { echo "other: $file" }
```

Both string and regex matches can be mixed. `else` is the fallback.
One-line and multi-line forms both work, same as `if` and `for`.

### Process Substitution

Feed command output as a file argument:

```
diff <(sort a.txt) <(sort b.txt)
wc -l <(find . -name "*.zig")
```

### Subshell Capture

Capture command output as a value:

```
branch = $(git branch --show-current)
count = $(ls *.zig | wc -l)
echo "On branch $branch with $count zig files"
```

### Heredocs and Herestrings

Slash replaces bash's `<<EOF` heredocs with triple-character delimiters.
No arbitrary token names. No `<<`. The delimiter itself is the signal.

**Three heredoc types:**

| Delimiter | Interpolation | Syntax highlighting |
|-----------|---------------|---------------------|
| `'''`     | No            | No                  |
| `"""`     | Yes           | No                  |
| `` ```lang ``  | Yes      | Yes, for `lang`     |

**Literal heredoc** (`'''`) — no variable expansion:

```
cat '''
    Hello $name
    This stays literal
    '''
```

Output is exactly `Hello $name` / `This stays literal` — no expansion.

**Interpolated heredoc** (`"""`) — variables expand:

```
name = "Steve"
cat """
    Hello $name
    Welcome to Slash
    """
```

Output: `Hello Steve` / `Welcome to Slash`.

**Syntax-highlighted heredoc** (`` ```lang ``) — interpolated, with
highlighting for the named language:

```
mysql $database ```sql
    SELECT * FROM users
    WHERE name = '$name'
    ORDER BY created_at DESC;
    ```
```

The shell highlights the heredoc body as SQL while you type. Useful for
embedded code in any language the highlighter supports.

**Indentation control** — the closing delimiter's indentation defines the
left margin. All content lines are dedented by that amount.

```
    cat '''
            line one
          line two
        '''
```

The closing `'''` is at column 8. So `line one` (12 spaces) keeps 4
leading spaces, and `line two` (10 spaces) keeps 2. You see the
indentation visually, and the closing delimiter tells the parser where
the left edge is.

**Piping** — the pipe can go on the opening line or after the closing
delimiter. Both forms are equivalent:

```
cat ''' | wc
    hello world
    '''

cat '''
    hello world
    ''' | wc
```

**Stacking** — multiple heredocs on one command line are matched in order:

```
diff <(cat ''') <(cat ''')
    file one
    '''
    file two
    '''
```

First `'''` matches the first block, second `'''` matches the second.

**Closing rule:** a line whose first non-whitespace characters are the
closing delimiter ends the heredoc. Anything after the delimiter on that
line continues the command. The leading whitespace before the delimiter
defines the margin.

**Herestring** (`<<<`) — feed a single string as stdin:

```
wc -w <<< "one two three"
grep "pattern" <<< $variable
tr a-z A-Z <<< "hello"
```

Variables are always interpolated in herestrings.
### User Commands (`cmd`)

User commands are the single mechanism for defining named commands in Slash.
There are no aliases, no functions — just `cmd`. One concept that handles
everything from simple shortcuts to multi-line logic.

Commands are not functions. There are no return values, no local scope, no
closures, no recursion. They are commands: give them a name, give them
arguments, they run in the current shell context.

**One-line form** — for simple shortcuts and one-liners:

```
cmd ll ls -la
cmd g git $*
cmd gco git checkout $*
cmd o open ${* ?? "."}
cmd mkcd(dir) mkdir -p $dir && cd $dir
cmd foo(one, two) ls $one $two $* 'caboose'
cmd ns (sudo lsof -nP -iTCP -sTCP:LISTEN || true) | sed -n '1p' | sort -sn
```

**Multi-line form** — for commands with logic:

```
cmd serve(port)
    port = $port ?? 8080
    python3 -m http.server $port

cmd deploy(env)
    if $env == ""
        echo "usage: deploy <env>"
        exit 1
    echo "Deploying to $env with args: $*"
    ssh $env "deploy $*"
```

**No quoting wrapper.** The body is everything after the name and optional
parameters. In multi-line form, the body is indentation-delimited. Single
quotes, double quotes, pipes — they are all just part of the command. No
escaping gymnastics. This is why aliases are unnecessary: `cmd` eliminates
the quoting problem entirely.

**Parsing rule:** `cmd name(params)` with `(` touching the name means a
parameter list. `cmd name body...` with a space means the body starts
immediately. This is unambiguous.

**Named parameters** capture positional arguments. `$*` inside a command
body refers to all arguments not captured by named parameters. `$#` is the
count of remaining arguments. If no `$*` or positional variables appear in
a one-liner body, `$*` is implicitly appended — so `cmd g git` and
`cmd g git $*` are identical.

Commands run in the shell's own process — `cd`, variable assignments, and
other state changes take effect in the current session. This is what makes
them different from external scripts.

Commands are defined in `slashrc` or any sourced `.slash` file. They compose
with everything: pipes, redirects, background, conditionals.

```
mkcd myproject
serve 3000 &
deploy staging --force
type mkcd                    # "mkcd is a user command"
```

**Managing commands** — `cmd` handles definition, inspection, deletion,
and listing. No extra keywords needed.

```
cmd foo echo hello           # define
cmd foo                      # show definition
cmd foo -                    # delete
cmd                          # list all user commands
```

**Command-not-found hook** — define `cmd ???` to handle unknown commands.
Slash calls it with the command name and arguments when nothing matches:

```
cmd ???(name)
    if test -f ./$name.rip
        rip ./$name.rip $*
        exit $?
    echo "slash: command not found: $name"
    exit 127
```

Three characters, visually says "unknown." If not defined, Slash prints the
default error message.

The hierarchy for name resolution: **builtins > user commands > PATH > `???`**.

### Auto-cd

If the input is a valid directory path but not a command, Slash `cd`s to it
automatically. No `cd` needed.

```
/tmp                       # same as: cd /tmp
~/projects                 # same as: cd ~/projects
..                         # already a builtin, but auto-cd handles paths too
```

Enabled by default. Disable with `set auto-cd false`.

### Default Values (`??`)

The `??` operator provides a default when a variable is unset or empty.
No distinction between "unset" and "empty" — `??` covers both.

**In assignments:**

```
port = $1 ?? 8080
name = $USER ?? "anonymous"
```

**Inline in command arguments:**

```
cmd code code -g ${* ?? "."}
cmd o open ${* ?? "."}
cmd hc head -c${1 ?? 12} $2
```

### Positional Arguments and Shift

| Variable | Meaning |
|----------|---------|
| `$1`, `$2`, ... `$9`, `${10}` | Individual positional arguments |
| `$*` | All arguments, or remaining arguments after named params |
| `$#` | Argument count |

`shift` discards `$1` and slides everything down — `$2` becomes `$1`, `$3`
becomes `$2`, etc. Useful when consuming arguments in a loop:

```
cmd flags
    while $# > 0
        if $1 == "-v"
            verbose = 1
        else
            echo "unknown: $1"
        shift
```

No `unshift`, `pop`, or `push`. Those are array operations. If you need
array manipulation, use a real language.

### Exiting a Command

`exit` is context-sensitive — it always means "leave the innermost context":

| Where | Effect |
|-------|--------|
| Inside a `cmd` body | Exits the command, returns to caller |
| Inside a sourced script | Exits the script, returns to caller |
| At the interactive prompt | Exits the shell |

Optional numeric argument sets the exit code (defaults to 0):

```
cmd mkcd(dir)
    if $dir == ""
        echo "usage: mkcd <dir>"
        exit 1
    mkdir -p $dir
    cd $dir
```

## Block Syntax: Braces vs Indentation

Slash supports both forms, everywhere, always.

**Brace form** — for one-liners at the interactive prompt:

```
if $x == 1 { echo one } else { echo other }
for f in *.zig { echo $f }
while $running { sleep 1 }
```

**Indentation form** — for scripts and multiline work:

```
if $x == 1
    echo one
else
    echo other

for f in *.zig
    echo $f

while $running
    sleep 1
```

**At the interactive prompt**, use `\` to continue onto the next line when using
indentation form:

```
$ for f in *.zig \
>     echo $f
```

**In `.slash` script files**, no `\` is needed. The parser knows it is reading a
file and treats indentation as block delimiters natively.

The lexer emits `LBRACE`/`RBRACE` tokens for the brace form and
`INDENT`/`OUTDENT` tokens for the indentation form. The grammar is identical
for both — `Block` accepts either. This is the same technique used in the
Rip language and proven to work cleanly.

## Job Control

Job control is implemented correctly. This is non-negotiable and is the
hardest part of the shell to get right. The implementation follows the POSIX
job control model precisely.

### Process Groups and Sessions

Slash is the session leader. Every pipeline runs in its own process group. This
ensures that `Ctrl+C` kills all processes in a pipeline, not just one.

```
ls | grep foo | wc -l
```

These three processes are in one process group. `Ctrl+C` sends `SIGINT` to the
entire group. All three die. Slash survives.

### Spawning a Foreground Job — Correct Sequence

This is the sequence that must be followed, exactly, every time:

```zig
const pid = try std.posix.fork();
if (pid == 0) {
    // Child process
    try std.posix.setpgid(0, 0);           // new process group
    try std.posix.tcsetpgrp(shell_tty, std.posix.getpid()); // take terminal
    resetSignalsToDefaults();
    try std.posix.execve(path, argv, envp);
} else {
    // Parent (Slash)
    try std.posix.setpgid(pid, pid);        // also set it — race condition prevention
    waitForJob(pid);
    try std.posix.tcsetpgrp(shell_tty, shell_pgid); // reclaim terminal
}
```

**Both parent and child call `setpgid`.** This is mandatory. You do not know
which runs first after `fork`. If only the child sets it, there is a window
where Slash might try to give the terminal to a process not yet in the right
group. Both set it. The second call is a no-op. The race condition is closed.

### Signal Handling

Slash's signal disposition:

| Signal    | Slash's disposition       | Child's disposition |
|-----------|--------------------------|---------------------|
| `SIGINT`  | Ignore (SIG_IGN)          | Default (terminate) |
| `SIGQUIT` | Ignore                    | Default             |
| `SIGTSTP` | Ignore                    | Default (stop)      |
| `SIGTTOU` | Ignore                    | Default             |
| `SIGTTIN` | Ignore                    | Default             |
| `SIGCHLD` | Handler (reap children)   | Default             |
| `SIGPIPE` | Ignore                    | Default (terminate) |

Children must have all signals reset to defaults before `exec`. Slash sets them
to ignore during initialization; children reset them before exec.

`SIGCHLD` is handled asynchronously. When a background job finishes, Slash
receives `SIGCHLD`, reaps the child with `waitpid(WNOHANG)`, updates the job
table, and prints `[1] Done  command` at the next prompt draw.

`SIGPIPE` in pipelines: when a reader closes its end (e.g., `head -5` got its
5 lines and exited), the writer receives `SIGPIPE` and terminates. This is
correct behavior. Slash ignores `SIGPIPE` itself; children get the default.

### Job Table

Slash maintains a job table. Each job has:

```zig
const Job = struct {
    id: u32,                    // job number shown to user: [1], [2]
    pgid: std.posix.pid_t,      // process group id
    state: JobState,            // Running, Stopped, Done
    exit_code: u8,
    command_text: []const u8,   // original command string for display
    processes: []Process,       // individual processes in the pipeline
};

const Process = struct {
    pid: std.posix.pid_t,
    state: ProcessState,
    exit_code: u8,
};
```

### Foreground and Background

```
make &             # start in background — Slash does not take terminal
fg                 # bring most recent background job to foreground
fg 2               # bring job [2] to foreground
bg                 # resume stopped job in background
bg 2               # resume job [2] in background
jobs               # list all jobs
```

`Ctrl+Z` sends `SIGTSTP` to the foreground process group. Slash receives control,
marks the job as Stopped, reclaims the terminal, and returns to the prompt.

`fg` sends `SIGCONT` to the stopped job's process group and transfers the
terminal back to it.

### Terminal Ownership Transfer

Transferring terminal ownership is done with `tcsetpgrp`. This must only be
called from the current foreground process or the session leader. Slash, as
session leader, always calls it. The sequence:

```
Ctrl+Z pressed:
  1. SIGTSTP delivered to foreground pgid
  2. Processes stop
  3. SIGCHLD delivered to Slash
  4. Slash calls tcsetpgrp(tty, slash_pgid)  -- reclaim terminal
  5. Slash redraws prompt

fg command:
  1. Slash calls tcsetpgrp(tty, job_pgid)  -- give terminal to job
  2. Slash calls kill(job_pgid, SIGCONT)   -- resume job
  3. Slash calls waitpid(job_pgid)         -- wait for it
  4. On return, Slash calls tcsetpgrp(tty, slash_pgid)  -- reclaim
```

## File Descriptor Handling

File descriptors are first-class. Slash supports the full set of Unix redirection
primitives.

### Standard Redirections

```
cmd > file          # stdout → file (truncate)
cmd >> file         # stdout → file (append)
cmd < file          # stdin ← file
cmd 2> file         # stderr → file
cmd 2>> file        # stderr → file (append)
cmd > file 2>&1     # stdout and stderr → file
cmd &> file         # stdout and stderr → file (shorthand)
cmd 2>&1 | other    # pipe stderr and stdout together
```

### Numbered File Descriptors

```
cmd 3> file         # fd 3 → file
cmd 3< file         # fd 3 ← file
cmd 1>&3            # stdout → fd 3
cmd 2>&3            # stderr → fd 3
```

### Pipelines and FD Wiring

For a pipeline `A | B | C`, Slash creates two pipes before forking:

```
pipe1: A_stdout → B_stdin
pipe2: B_stdout → C_stdin
```

After forking each process:
- A: close read end of pipe1, close both ends of pipe2, dup pipe1 write → stdout
- B: close write end of pipe1, close read end of pipe2, dup pipe1 read → stdin, dup pipe2 write → stdout
- C: close both ends of pipe1, close write end of pipe2, dup pipe2 read → stdin

All unused pipe ends are closed before exec. Failure to close them causes
`SIGPIPE` never to be delivered and pipelines to hang.

### Process Substitution

```
diff <(sort a.txt) <(sort b.txt)
```

Implemented with named pipes (FIFOs) or `/dev/fd/N`. Slash creates a pipe, forks
a child to run the substituted command writing to the pipe's write end, and
passes the read end's path (`/dev/fd/3`) as an argument to the outer command.

## Tab Completion

Tab completion is fast, accurate, and context-aware.

- **Command position:** complete against PATH, builtins, and user commands
- **Argument position:** complete against files and directories by default
- **Flag position:** complete against known flags for common commands (git,
  zig, cargo, npm, etc.) — completions are defined in a completions directory
- **Variable position:** complete against defined variables and environment
- **After `cd`/`j`:** complete against directories only

Completion uses the actual parser for context — it knows whether the cursor
is in command position, argument position, or inside a string. Completions
are never wrong about context.

Completion definitions for external commands live in
`~/.config/slash/completions/`. They are simple Slash scripts that register
completion functions. A standard library of completions ships with Slash.

## Key Bindings

Key bindings use the `key` keyword with human-readable combo names. No
`\e[3~` escape sequences, no `${terminfo[...]}` indirection.

**Syntax:** `key <combo> <action-or-command>`

If the value is a bare word, it is a readline action. If it is a quoted
string, it is executed as a command.

**Default bindings** — these ship with Slash and work out of the box:

```
# Navigation
key esc-=       dirs             # interactive directory picker
key esc-1       ".."             # up 1 directory
key esc-2       "..."            # up 2 directories
key esc-3       "...."           # up 3 directories
key esc-4       "....."          # up 4 directories
key esc-5       "......"         # up 5 directories
key esc--       "cd -"           # previous directory
key esc-l       "ls -la"         # quick directory listing

# History
key ctrl-r      history-search   # incremental history search
key up          history-prefix-up
key down        history-prefix-down
key esc-p       history-prefix-up
key esc-n       history-prefix-down

# Editing
key home        beginning-of-line
key end         end-of-line
key ctrl-b      backward-word
key ctrl-f      forward-word
key backspace   delete-backward
key delete      delete-forward
key ctrl-w      kill-word-backward
key ctrl-x-e    edit-in-editor   # open current line in $EDITOR
key shift-tab   complete-prev
```

**Key combo notation:**

| Notation | Meaning |
|----------|---------|
| `ctrl-x` | Control + x |
| `esc-x` | Escape then x (or Alt+x) |
| `ctrl-x-e` | Control+x followed by e |
| `up`, `down`, `home`, `end` | Arrow and navigation keys |
| `pageup`, `pagedown` | Page keys |
| `shift-tab` | Shift + Tab |
| `backspace`, `delete` | Delete keys |

**Custom bindings** in `slashrc`:

```
key esc-g       "git status"
key esc-t       "tree -C -L 2"
key ctrl-x-d    dirs
```

The `dirs` action is interactive: it displays the recent directory list
below the prompt, waits for a digit, and jumps immediately. No Enter
needed. Press Escape or any non-digit to cancel.

## History

History is stored in SQLite at `~/.local/share/slash/history.db`.

Schema:

```sql
CREATE TABLE history (
    id        INTEGER PRIMARY KEY,
    command   TEXT NOT NULL,
    cwd       TEXT NOT NULL,
    exit_code INTEGER NOT NULL,
    duration  INTEGER NOT NULL,  -- milliseconds
    ts        INTEGER NOT NULL   -- unix timestamp
);

CREATE INDEX history_command ON history(command);
CREATE INDEX history_ts ON history(ts);
```

**Interactive search:** `Ctrl+R` opens an incremental history search. Results
are ranked by recency and frequency. The full command is shown with its
working directory and exit code.

**History is shared** across all running Slash instances. When a command
completes in any Slash session, it is immediately available in all others.

**Deduplication:** consecutive identical commands are stored once. `ls` typed
ten times in a row is one history entry.

**Privacy:** commands starting with a space are not recorded in history.
Useful for sensitive commands. Enabled by default.

```
 mysql -p secretpassword          # leading space = not recorded
```

Disable with `set history-ignore-space false`.

---

## Directory Frecency

Stored in SQLite at `~/.local/share/slash/history.db`.

Schema:

```sql
CREATE TABLE dirs (
    id        INTEGER PRIMARY KEY,
    path      TEXT UNIQUE NOT NULL,
    visits    INTEGER NOT NULL DEFAULT 1,
    last_seen INTEGER NOT NULL  -- unix timestamp
);
```

Frecency score: `visits / (1 + seconds_since_last_visit / 86400.0)`

The `dirs` action (bound to `Esc-=` by default) shows the top 10 directories
by frecency score. The `j` command does a fuzzy match against directory path
components and picks the highest-scoring match.

## Configuration

Slash configuration lives in `~/.config/slash/slashrc`. It is a Slash script that
runs at startup. It can set variables, define commands, and set shell options.

```
# ~/.config/slash/slashrc

# Environment
EDITOR = "vim"
PAGER = "less"

# Commands
cmd ll ls -la
cmd g git $*
cmd gco git checkout $*
cmd o open ${* ?? "."}
cmd mkcd(dir) mkdir -p $dir && cd $dir

cmd serve(port)
    port = $port ?? 8080
    python3 -m http.server $port

# Options (set/show/reset follow the cmd pattern)
set history-limit 10000    # set an option
set prompt-git true        # set prompt-git              # show current value
set prompt-git -           # reset to default
set                        # list all options
```

No plugin manager. No framework. No oh-my-slash. If you want something, write
it in Slash or call a program.

## Build Order

Build Slash in this order. Each layer must be solid before the next begins.

**Phase 1 — Job Control Core**

Build a standalone Zig program (not yet a shell) that correctly:
- Forks a foreground job and waits for it
- Handles `Ctrl+C` without dying
- Handles `Ctrl+Z` to stop a job and reclaims the terminal
- Resumes a stopped job with `fg`
- Runs a background job with `&` and reports completion
- Handles `SIGPIPE` correctly in a two-process pipeline

This is the foundation. It must be correct. Test it exhaustively before
proceeding. The reference for this work is the GNU libc manual chapter
"Implementing a Job-Control Shell."

**Phase 2 — Pipeline and Redirection**

Wire the job control core into a pipeline executor:
- Fork and exec a single command
- Build multi-stage pipelines with correct pipe FD wiring
- Implement all redirect forms
- Implement process substitution
- Close all unused FDs before exec

**Phase 3 — Parser and Expander**

- Integrate the Zig grammar engine with the Slash grammar
- Implement the lexer (tokenizer)
- Parser outputs s-expressions directly — no AST construction
- Implement glob expansion (in-process — no shell glob failures)
- Implement variable expansion
- Implement inline math evaluation

**Phase 4 — Builtins and Eval**

- Implement the full builtin set
- Walk s-expressions recursively and dispatch to builtins or external commands
- Implement if/unless, for, while/until, try with correct exit code semantics
- Implement `cmd` definitions (one-line and multi-line), `cmd ???` hook
- Implement semicolons, subshell grouping, auto-cd
- Implement `??` default values, `set` option management

**Phase 5 — Readline, Key Bindings, and Highlighting**

- Implement line editing (or integrate a minimal readline)
- Implement `key` binding system with human-readable combo notation
- Implement default key bindings (navigation, history, editing)
- Implement `dirs` interactive directory picker (triggered by hotkey)
- Implement live syntax highlighting using the parser
- Implement multi-line editing with `\` continuation at the prompt
- Implement indentation-based blocks in script mode

**Phase 6 — SQLite Layer**

- Implement history storage and retrieval
- Implement `Ctrl+R` incremental search
- Implement directory frecency tracking
- Implement directory frecency ranking and `j` (fuzzy jump)

**Phase 7 — Prompt**

- Implement async prompt rendering
- Git status (non-blocking)
- Language version detection
- Duration and exit code display

**Phase 8 — Completion**

- Context-aware completion engine
- File/directory completion
- Command completion against PATH
- Variable completion
- Completion definition format and standard library

## What Slash Is Not

Slash is not a POSIX shell. It does not aim for POSIX compliance. It will not
run bash scripts. It does not need to. If you need to run a bash script, call
bash. Slash is your interactive shell and your scripting language for tasks that
are genuinely shell tasks. Everything else calls out to the right tool.

Slash is not a programming language. It has commands, not functions — no return
values, no local scope, no closures, no recursion. It has no data structures
(use jq or a real language). It has no modules or imports. It does not pretend
to replace Python, TypeScript, or Rip. It composes with them.

Slash is not configurable to the point of chaos. There is no plugin system. There
is no theme engine. There is no package manager for shell extensions. Slash ships
with good defaults and a simple configuration file. That is enough.

**Slash** — the fundamental operator of every shell. Paths, options, roots,
