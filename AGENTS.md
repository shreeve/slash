# AGENTS.md — Working on Slash

> Slash is a Unix shell with structured commands, composable pipelines,
> and first-class jobs — one grammar, end to end.

This file is for agents (human or AI) modifying the Slash codebase.
Read it before editing.

## The one rule

> **Does this improve `Command` clarity, `Pipeline` correctness, `Program`
> composability, or `Job` control? If not, do not build it.** (PLAN §14)

This is the test every feature, refactor, and design choice has to pass.
If a proposal can't be honestly answered "yes" on at least one of those
four axes, it doesn't ship. The PLAN's principle is what keeps Slash a
shell instead of drifting into a programming language, a data shell, or
an AI shell.

## Architecture

Four-stage pipeline:

```
source  →  Shape  →  Program  →  Job
         (parse)    (lower)      (run)
```

| Stage   | Layer       | Responsibility                                       |
|---------|-------------|------------------------------------------------------|
| Source  | bytes       | Raw user input                                       |
| Shape   | syntactic   | Parsed structure, span-bearing, source-faithful      |
| Program | semantic    | Lowered, immutable, executable                       |
| Job     | runtime     | Mutable runtime state, process group, result         |

`PLAN.md` is the single source of truth for the design. Read sections
§1–§7 before substantive changes.

## File ownership

- **`slash.grammar`** — the grammar DSL. The single source of truth for
  surface syntax. **Edit this; never edit `src/parser.zig`.**
- **`src/parser.zig`** — generated by [nexus](https://github.com/shreeve/nexus)
  from `slash.grammar`. Regenerate with
  `/Users/shreeve/Data/Code/nexus/bin/nexus slash.grammar src/parser.zig`.
  Commit grammar and generated parser together.
- **`src/slash.zig`** — hand-written language module: `Tag` enum,
  `keywordAs` keyword promotion, the `Lexer` wrapper (comment skipping,
  `NAME_EQ` fusion, `INDENT`/`OUTDENT` virtual tokens, heredoc body
  collection).
- **`src/shape.zig`** — Shape types and the strict Sexp → Shape
  converter.
- **`src/word.zig`** — `Word` lowering and quote/escape decoding.
- **`src/program.zig`** — Program kernel forms and Shape → Program
  lowering.
- **`src/eval.zig`** — the only module that knows shell semantics.
  Expansion, control flow, builtin dispatch.
- **`src/exec.zig`** — POSIX plumbing only. No shell policy.
- **`src/job.zig`** — `Job` / `JobTable`, monotonic state graph, wait
  service.
- **`src/builtins.zig`** — builtin registry and implementations.
- **`src/session.zig`** — long-lived shell state.
- **`src/vars.zig`** — `VarStore`: scalar and list variables, exported
  flag.
- **`src/runtime.zig`** — `Result`, `Signal`, status-byte conversion.
- **`src/diagnostics.zig`** — `Diagnostic`, `Sink`, error codes.
- **`src/headless_tests.zig`** — end-to-end integration tests.
- **`src/pty_tests.zig`** — PTY-driven REPL tests; spawns the built
  `bin/slash` against a pseudo-terminal and asserts on rendered output.
  Run via `zig build test-pty` (also runs as part of `zig build test`).
- **`src/main.zig`** — CLI entry point.
- **`PLAN.md`** — design constitution.
- **`ROADMAP.md`** — concrete remaining work; items disappear as they
  ship.

## Critical invariants

These survive every commit. If a change forces breaking one, the change
is wrong.

1. **`source → Shape → Program → Job` is the only execution path.** No
   shortcuts.
2. **`Program` is immutable after lowering.** All runtime mutation lives
   in `Job` / `Session`.
3. **`argv` is never flattened to a shell string.** The exec input is
   `exe: []const u8` plus `argv: []const []const u8`.
4. **Expansion happens once per word.** `Word → argv` is a one-way gate.
   No re-scanning argv elements for variables, command substitutions, or
   globs.
5. **`pipefail = on` by default.** No surface syntax toggles it.
6. **No implicit word-splitting after expansion.** Scalars stay one
   argument; lists splice as N arguments. Quoting prevents globbing and
   preserves field boundaries.
7. **Brace blocks and indentation blocks are the same syntax.** `if cmd
   { body }` and `if cmd\n  body` produce the same Shape.
8. **One grammar, one parser.** No regex-based completion or
   highlighting hacks. The grammar drives every text-shape concern.
9. **Job state is monotonic.** `pending → running → {stopped↔running}*
   → done`. Once `done`, it stays `done`.
10. **Builtins never call `exec`.** A shell-context builtin's Job has
    `processes.len == 0`.

## Coding conventions

- **Zig 0.16.** `std.posix.fork`/`waitpid`/`pipe`/`close`/`dup2`/`_exit`
  were removed; use `std.c.*` with `std.c.errno(rc)`. See
  `ZIG-0.16.0.md`.
- **No temporal framing.** Don't write "currently does X but later will do
  Y", "deferred to phase 2", "TODO add this", or "for now we just". Code
  describes what *is*. Features that haven't shipped yet simply aren't
  referenced. The ROADMAP is where future work lives.
- **Comments explain non-obvious intent, trade-offs, or constraints.**
  Don't narrate what the code does line-by-line.
- **Public-API names are real English words a Unix engineer
  understands.** `Shape`, `Word`, `Command`, `Pipeline`, `Program`,
  `Sequence`, `Job`, `Redirect`. **Forbidden**: `ExecPlan`,
  `InvocationSpec`, `ProcessNode`, `ASTNode`, `Stmt`, `Expr`, `Node`, any
  half-name. (PLAN §5.)
- **Use the existing `Diagnostic` infrastructure for errors.** Codes are
  `SHxxxx` (shape), `LWxxxx` (lower), `EVxxxx` (eval), `EXxxxx` (exec),
  `JBxxxx` (job).

## Build and test

Requires Zig 0.16.0.

```sh
zig build                                # build ./bin/slash
zig build run -- --version               # version
zig build test                           # run all tests (headless + PTY)
zig build test-pty                       # run only the PTY-driven REPL tests
zig build run -- -c 'echo hello'         # one-liner
zig build run -- script.sh arg1 arg2     # script with positional args
```

After grammar changes:

```sh
/Users/shreeve/Data/Code/nexus/bin/nexus slash.grammar src/parser.zig
zig build test
```

## Working on items from the ROADMAP

When you ship an item from `ROADMAP.md`, **delete it from that file** in
the same commit. The ROADMAP shrinks until empty; that's how 1.0 ships.

For each item:
1. Read the item's description.
2. Verify it still passes the §14 test. (It should, but always re-check
   against the rule above.)
3. Look at related PLAN.md sections.
4. Implement, test, update ROADMAP.

## When in doubt

Re-read PLAN §14. If the change doesn't earn its place by improving
`Command`, `Pipeline`, `Program`, or `Job`, it doesn't go in.
